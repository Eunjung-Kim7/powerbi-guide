데이터 정제·변환·결합 · 3~7주차
▶
📖 핵심 용어 정의
파워쿼리 편집기
데이터를 시각화 전에 정제·변환하는 ETL 도구. 모든 작업이 적용된 단계(Applied Steps)로 기록 → 언제든 수정·삭제 가능. M 언어 기반.
적용된 단계 (Applied Steps)
오른쪽 패널에 변환 작업이 순서대로 쌓임. 단계 클릭 → 그 시점 데이터 상태 확인 가능. 잘못된 단계는 X 클릭으로 삭제.
피벗 (Pivot)
행 값을 열로 펼치기. 연도가 행으로 나열된 데이터를 연도별 열로 변환할 때 사용.
예) 연도 행 → 2022열·2023열·2024열
언피벗 (Unpivot)
여러 열을 하나의 행으로 쌓기. 공공데이터의 연도별 열 구조를 Power BI에 맞는 행 구조로 변환할 때 필수.
예) 2022열·2023열 → 연도·값 두 열
추가 (Append)
같은 구조 테이블을 세로로 합치기 (SQL UNION).
예) 2023년 판매 + 2024년 판매 → 전체 판매
병합 (Merge)
공통 키로 두 테이블을 가로로 합치기 (SQL JOIN). 왼쪽 외부·내부·완전 외부 조인 지원.
예) 판매 + 고객 테이블 → 고객ID로 병합
null
값이 없음을 나타내는 특수값. 빈 셀과 다름. 처리법: 변환 → 채우기(아래) / 홈 → 행 제거 → 오류 행 제거 / 값 바꾸기
데이터 형식
반드시 정확히 설정. DAX 함수·관계·날짜 인텔리전스 정상 작동 조건.

🗂️ 데이터 모델이란? — "엑셀 VLOOKUP을 자동화한 것"
엑셀에서 두 시트를 연결할 때 VLOOKUP을 쓰죠? Power BI의 데이터 모델은 그 연결을 한 번만 설정해두면 이후에는 자동으로 작동하는 구조입니다.
예) 세대 테이블과 자치구 테이블을 "자치구명"으로 연결 → 이후 시각화에서 두 테이블 데이터를 자유롭게 함께 사용 가능.
🔗 관계 설정 — 클릭 경로와 핵심 개념
관계 설정하는 방법
Model 뷰(왼쪽 세 번째 아이콘) 클릭 →

기본 키(PK) vs 외래 키(FK)
기본 키(PK): 중복 없는 고유값 열. 예) 자치구 테이블의 "자치구코드"
외래 키(FK): 다른 테이블을 참조하는 열. 예) 세대 테이블의 "자치구코드"

👉 PK 쪽이 "1", FK 쪽이 "다(N)" — 항상 1:N 방향으로 연결
1:N 관계란? — 비유로 이해하기
자치구 테이블 (1쪽): 강남구, 관악구, 마포구… → 각 이름이 딱 1번씩만 나옴
세대 테이블 (N쪽): 강남구 2020년, 강남구 2021년… → 같은 자치구가 여러 번 나옴

👉 "1"인 테이블(자치구)로 "N"인 테이블(세대)을 필터링하는 방향이 기본
⚙️ 관계 속성 — 관계 선을 더블클릭하면 보이는 설정들
속성 쉬운 설명 & 언제 쓰나
활성 관계
실선		기본으로 사용되는 관계. 두 테이블 사이에 실선으로 표시. 자동으로 필터가 전달됨. 대부분의 경우 이것 하나만 쓰면 됨
비활성 관계
점선		같은 테이블 사이에 두 번째 관계를 만들 때 자동으로 비활성화됨(점선). DAX에서 USERELATIONSHIP()으로만 사용 가능
필터 방향
단방향		기본값. 권장. 1→N 방향으로만 필터 전달. 예) 자치구 필터 → 세대 데이터 필터링. 반대 방향은 자동으로 안 됨
필터 방향
양방향		양쪽 모두 필터 전달. 복잡한 모델에서는 의도치 않은 필터 전파로 오류 발생 가능 → 꼭 필요한 경우에만 사용
카디널리티관계의 종류. 1:N(가장 일반적) / 1:1(같은 키가 양쪽에 딱 하나씩) / N:N(중간 브릿지 테이블 필요, 고급)
🔑 다중 키 — 두 열 이상으로 연결해야 할 때
다중 키란?
Power BI 관계는 열 하나로만 연결 가능합니다. 하지만 실무에서는 "자치구+연도" 조합으로만 유일한 경우가 있어요.

예) 세대 테이블에서 강남구-2023이 유일한 행 → "자치구코드"만으로는 중복 발생
해결 방법 — 복합 키 열 만들기
파워쿼리 또는 계산 열로 두 값을 합친 새 열 생성 후 그 열로 관계 연결:

파워쿼리: 열 추가 → 사용자 지정 열
[자치구코드] & "_" & Text.From([연도])

계산 열(DAX):
복합키 = [자치구코드] & "_" & FORMAT([연도],"0")
📋 계산 열 (Calculated Column) — "테이블에 새 열 추가"
언제 씁니까?
• 행마다 다른 값을 계산해서 테이블에 저장할 때
• 슬라이서 기준으로 쓸 분류값을 만들 때

🔢 숫자 범주화 (Binning) — 연속 숫자를 구간으로 분류
💡 Bin이란?
Bin은 영어로 "통, 상자, 구간"이라는 뜻입니다. 데이터 분석에서는 연속적인 숫자를 일정 간격의 구간(통)으로 나눠 담는 것을 의미합니다.

예를 들어 1인 세대 수가 3,200 / 7,800 / 14,500 / 22,000 처럼 다양한 숫자로 흩어져 있을 때,
→ "0~5000", "5000~10000", "10000~20000", "20000 이상" 구간(Bin)으로 나누면
→ 분포 패턴을 한눈에 파악할 수 있게 됩니다.

히스토그램 차트의 각 막대 하나가 바로 Bin 하나입니다.
연속적인 숫자 데이터(세대 수, 비율 등)를 일정 구간(Bin)으로 나눠 분포를 파악하거나 히스토그램을 만들 때 사용합니다.

앞에 ①②③④ 번호를 붙이면

💡 그룹 기능 vs 계산 열: 그룹(Bin)은 구간 크기가 일정할 때 빠르게 사용. 계산 열(SWITCH)은 구간 경계를 직접 지정하거나 앞에 번호를 붙여 정렬 순서를 제어할 때 유리.
📅 날짜 테이블 — 시간 분석의 필수 준비물
왜 필요한가? TOTALYTD, SAMEPERIODLASTYEAR 같은 시간 인텔리전스 함수는 연속된 날짜 테이블이 있어야 작동합니다. 없으면 함수가 오류를 냅니다.

만드는 방법 (2가지):
① 자동 생성: 홈 탭 → 새 테이블 → 날짜테이블 = CALENDARAUTO() → 데이터의 최소·최대 날짜 기준 자동 생성
② 범위 지정: 날짜테이블 = CALENDAR(DATE(2019,1,1), DATE(2024,12,31))

생성 후 반드시: 날짜 테이블의 날짜 열 선택 → 테이블 도구 탭 → 날짜 테이블로 표시 클릭

DAX 개념 이해
What is DAX?
DAX란?
Data Analysis Expressions의 약자로, Analysis Services · Power BI · Excel 파워피벗에서 사용하는 수식 언어입니다.

DAX 수식은 테이블이나 열을 참조하여 계산식을 작성하며, 다양한 분석 함수·연산자·값이 포함됩니다.
-- 기본 구조 측정값명 = 함수명(테이블명[열이름])
DAX 구문
기본 구문 규칙
• 테이블 참조: 테이블명[열이름]
• 측정값 참조: [측정값명]
• 문자열: 큰따옴표 "텍스트"
• 주석: -- 한 줄 또는 /* 여러 줄 */
-- 열 참조 예시 총매출 = SUM(판매[매출금액]) -- 측정값 참조 예시 매출비율 = DIVIDE([총매출], [전체매출])
DAX 연산자
연산자 종류
산술: + - * / ^
비교: = <> > < >= <=
텍스트 연결: &
논리: && || IN NOT
-- 텍스트 연결 지역구분 = 지역[시도] & " " & 지역[구군] -- 논리 연산자 서울여부 = 지역[시도] = "서울특별시"
측정값 vs 계산 열 vs 계산 테이블
구분 📐 측정값 (Measure) 📋 계산 열 (Calculated Column) 🗂️ 계산 테이블
계산 시점 보고서에서 사용할 때 동적으로 계산 모델 로드 시 미리 계산 후 저장 모델 로드 시 테이블 전체 생성
저장 위치 메모리에서 계산 (저장 안 됨) 테이블에 새 열로 저장 새 테이블로 저장
컨텍스트 필터 컨텍스트에 반응 (슬라이서·필터 적용) 행 컨텍스트 (각 행마다 계산) 테이블 전체 컨텍스트
주요 사용처 합계, 평균, 비율, 누계 등 집계 분석 텍스트 결합, 날짜 추출, 분류 기준 등 행 단위 계산 날짜 테이블, 필터 보조 테이블 생성
DAX 예시
총매출 = SUM(판매[매출금액])
연도 = YEAR(판매[날짜])
날짜테이블 = CALENDARAUTO()
시각화 사용 값 영역에 배치 축·범례·행에 배치 관계 설정 후 활용
💡 선택 기준: 숫자 집계가 필요하면 측정값, 행마다 다른 값을 열로 추가해야 하면 계산 열. 측정값이 성능상 유리하므로 가능하면 측정값 우선 사용.

관계 설정이 안 됨 / 카디널리티 오류
Model 뷰에서 선을 연결했는데 오류 메시지가 뜨는 경우
"이 관계를 만들 수 없습니다" 또는 카디널리티가 예상과 다르게 설정되는 오류.
원인별 해결
원인 1
연결하려는 열에 중복값이 있음 (N:N 오류)
1쪽(차원 테이블)에 중복값이 있으면 관계가 다대다(N:N)로 설정됨. 차원 테이블의 키 열에 중복 여부 확인.
확인: Data 뷰 → 해당 열 클릭 → 열 도구 탭 → "고유 값" 확인. 중복 있으면 파워쿼리에서 중복 제거(홈 → 행 제거 → 중복 행 제거)
원인 2
두 열의 데이터 형식이 다름
한 쪽은 텍스트(자치구코드 "110"), 다른 쪽은 숫자(110)이면 연결 안 됨.
해결: 파워쿼리에서 두 열 모두 같은 형식(텍스트 또는 정수)으로 변환 후 재연결
원인 3
이미 관계가 존재해서 순환 오류
두 테이블 사이에 이미 다른 경로로 관계가 연결되어 있어 순환 발생.
해결: 홈 탭 → 관계 관리에서 기존 관계 확인 후 불필요한 관계 삭제

STEP 1

파일 → 옵션 및 설정 → 옵션 → 보안 → "맵 및 등치 지역도 시각적 개체" 체크(✓) 확인 → 확인 → Power BI 완전히 재시작
STEP 2
자치구명 열에 데이터 범주 설정

STEP 3
값이 "강남구"인지 "강남"인지 확인
Power BI 맵은 "강남구", "관악구"처럼 "구"가 붙은 공식 행정 명칭을 사용해야 인식. "강남", "관악"처럼 줄여쓰면 오류 발생

문제 3
팀원 PC에서 파일 열면 데이터 연결 오류
해결: 데이터 파일(Excel·CSV)을 팀 공유 폴더에 고정 경로로 저장. 파일 → 옵션 및 설정 → 데이터 원본 설정에서 경로 변경 가능. OneDrive·Google Drive 공유 폴더 활용 권장.
Tip
작업 전 백업 습관화
큰 변경 전 반드시 다른 이름으로 저장(Ctrl+Shift+S). Power BI는 실행 취소(Ctrl+Z)가 제한적이어서 작업 전 복사본 저장이 가장 안전한 버전 관리입니다.
가장 자주 발생하는 문제 — 우선순위별 해결
📅
열 병합 후 날짜 서식 지정 시 모든 행 오류
엑셀 데이터 → 파워쿼리 열 병합 → 날짜 변환 시 발생 · 로캘 한국어 설정에서도 오류
엑셀에서 가져온 데이터를 파워쿼리에서 열 병합하면 결과값은 텍스트로 만들어집니다. 이 텍스트를 날짜로 바꿀 때 눈에 보이지 않는 공백·제어문자가 섞여 있거나, 변환 방식이 잘못되면 모든 행이 오류가 됩니다. 아래 순서대로 따라하세요.
단계별 해결 방법 — 클릭 경로 포함
STEP 0
사전 점검 — Windows 지역 형식 확인
Power BI 옵션에서 "Windows 기본 표시 언어 사용"으로 설정된 경우, Windows 지역 형식이 영어(미국)로 되어 있으면 날짜를 MM/DD/YYYY로 읽어 오류 발생.

확인 경로:
Windows 설정 → 시간 및 언어 → 언어 및 지역 → 지역 형식: 한국어(대한민국) 확인

Power BI에서 직접 지정:
파일 → 옵션 및 설정 → 옵션 → 현재 파일 → 지역 설정 → 한국어(대한민국) 선택
STEP 1
병합된 열 선택
파워쿼리 편집기에서 날짜로 만든 병합 열의 헤더를 클릭하여 선택합니다.
열 헤더 왼쪽 아이콘이 ABC(텍스트)인지 확인하세요. 텍스트가 맞으면 다음 단계로 진행합니다.
STEP 2
공백·제어문자 제거 — 트리밍(Trim) + 정리(Clean)
상단 메뉴 변환 탭 클릭 → 형식 버튼 클릭 → 드롭다운에서 다듬기(Trim) 클릭
→ 앞뒤 공백 제거 완료

같은 방법으로 변환 탭 → 형식 → 정리(Clean) 클릭
→ 줄바꿈·탭·제어문자 제거 완료

⚠️ 변환 탭이 보이지 않으면 열을 먼저 선택해야 활성화됩니다
STEP 3
날짜로 형식 변환 — 반드시 "로캘 사용"으로
열 헤더 우클릭 → 형식 변경 → 로캘 사용 클릭
→ 창이 뜨면:
  • 데이터 형식: 날짜 선택
  • 로캘: 한국어(대한민국) 선택
  • 확인 클릭

💡 "현재 로캘 사용"이 아닌 "로캘 사용"을 선택해야 명시적으로 한국어 날짜 형식을 지정할 수 있습니다
STEP 4
STEP 3까지 해도 오류라면 — M 코드로 직접 변환
상단 수식 입력줄이 안 보이면: 보기 탭 → 수식 입력줄 체크

오른쪽 적용된 단계 패널에서 날짜 변환 단계 클릭 → 수식 입력줄에 아래 코드 입력:
-- 2024-01-01 형태 텍스트 → 날짜 변환 = Table.TransformColumnTypes( 이전단계, {{"날짜열이름", type date}}, "ko-KR" )
• 이전단계: 수식 입력줄에 원래 있던 단계 이름으로 교체
• 날짜열이름: 실제 열 이름으로 교체 (따옴표 포함)
STEP 5
여전히 안 되면 — 오류 행 확인 후 값 형태 점검
열 헤더 필터 아이콘(▼) 클릭 → "오류" 항목 체크 → 확인
→ 오류가 나는 실제 값이 보입니다. 값 형태를 확인하세요.

자주 보이는 문제:
  • 빈 셀(null) 포함 → 홈 탭 → 행 제거 → 오류 행 제거
  • 일부 셀이 다른 형식(예: "2024년 1월 1일") → 해당 행만 찾아 수동 정리
  • 열 병합 시 구분자가 잘못 들어간 경우 → 병합 단계로 돌아가 재설정

📦 데이터 생태계 — Power BI 너머의 세계
Power BI는 데이터 시각화·분석 도구입니다. 하지만 실무에서는 데이터가 시각화 전에 수집 → 정제 → 가공 → 저장되는 데이터 파이프라인 전 과정이 먼저 필요합니다. 이를 담당하는 것이 데이터 엔지니어링입니다.
🏭 데이터 팩토리 (Data Factory)
데이터 수집·정제·가공·분석 자동화 시스템
사람 개입 없이 데이터 흐름을 자동 처리. 자동화·확장성·재사용성·실시간성이 핵심.
1데이터 수집 — 로그·DB·API·IoT
2데이터 정제 — 오류·중복 제거·형식 통일
3데이터 변환 — 집계·필터링·가공
4데이터 저장 — 웨어하우스·레이크
5데이터 활용 — 대시보드·AI·리포트
대표 도구
Azure Data Factory Apache Airflow Google Cloud Dataflow
⚙️ ETL vs ELT 파이프라인
데이터 이동의 두 가지 방식
ETL (Extract → Transform → Load)
소스 추출 → 변환 먼저 → 저장. 전통적 방식. 온프레미스 환경.
ELT (Extract → Load → Transform)
소스 추출 → 저장 먼저 → 변환 나중에. 클라우드 환경의 주류. 빅데이터에 유리.
Power BI 파워쿼리는 소규모 ETL 역할. 대용량 자동화엔 Data Factory 같은 전용 도구 필요.
🗄️ 데이터 저장소 유형
데이터가 쌓이는 곳의 종류
데이터 웨어하우스 (DW)
정형 분석용 저장소. Snowflake, BigQuery, Redshift. Power BI와 직접 연결 가능.
데이터 레이크 (Data Lake)
비정형 포함 원시 데이터 저장. Azure Data Lake, AWS S3. 분석 전 가공 필요.
Microsoft Fabric (2024~)
Data Factory + 웨어하우스 + Power BI 통합 플랫폼. Power BI Premium에 포함.
